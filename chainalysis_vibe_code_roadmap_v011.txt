CHAINALYSIS-LIKE TOOL — VIBE CODE ROADMAP (v0.0.1.1)
=====================================================

PHILOSOPHY
----------
Build incrementally. Each step produces a working app you can use and demo.
No full nodes. No Kafka. No Spark. Use public APIs and managed services.
Ship fast, replace parts later when scale demands it.


REEVALUATED TECH STACK
----------------------

Frontend:       React (Vite) + Tailwind CSS
                TanStack Query (React Query) — ALL API fetching, caching, async enrichment
                Zustand — global UI state (trace session, filters, renderer lock)
Visualization:  Cytoscape.js (<300 nodes) + Sigma.js WebGL (300+ nodes, locked at trace start)
                Recharts (charts/timelines) — BigInt → Number cast for data arrays
                Web Workers for layout computation (dagre/COSE offloaded from main thread)
Backend:        Python FastAPI (simple, async, great ecosystem)
                SQLAlchemy 2.0 Async (Declarative Base) — ALL database access
                Alembic — ALL schema creation and migrations (no raw CREATE TABLE)
                + slowapi (inbound rate limiting on heavy endpoints)
                + aiolimiter (proactive outbound rate limiting — global token bucket)
                + SSE via sse-starlette (streaming trace progress to frontend)
Database:       SQLite (with WAL mode + aiosqlite) → PostgreSQL → Neo4j/Memgraph
                (start flat, migrate relational when needed, graph DB when
                recursive CTEs bottleneck N-hop traces at scale)
Blockchain:     Public APIs only (no nodes)
                - Blockstream.info API (Bitcoin, free, no key)
                - Etherscan / Basescan / etc. (EVM chains, free tier)
                  ↳ txlist (native), tokentx (ERC-20), txlistinternal (internal)
                - Blockchain.com API (Bitcoin, free)
                - Solscan / Helius (Solana)
                - LayerZero / Wormhole APIs (cross-chain bridge parsing)
Prices:         CoinGecko free API (current AND historical price at time of tx)
Labels/Tags:    Etherscan label DB (scraped/public), Arkham public data,
                Walletexplorer.com (BTC clusters), OFAC SDN list (free CSV)
Decoding:       4byte.directory API / eth-abi (function signature decoding for EVM)
Precision:      Backend: Python Decimal for all value calculations
                Frontend: BigInt or ethers.js utils for all wei/satoshi math
Testing:        pytest + vcrpy/responses (record & replay HTTP for offline CI)
Deployment:     Docker Compose locally, Railway/Fly.io when ready
Secrets:        .env file from day one (see ENVIRONMENT SETUP below)

  ⚠️  BUG PREVENTION: SQLite WAL Mode on Docker Volumes (Corruption Risk)

  ENV-6  [x] SQLite's WAL mode relies on shared memory files (-shm, -wal) for
               concurrent reader/writer coordination. These shared memory locks
               FAIL or CORRUPT on networked filesystems and certain virtualized
               OS mounts — which is exactly what Docker Desktop for Windows/Mac
               uses for managed volumes.

               RULES:
               - LOCAL DEV (Docker Desktop on Mac/Windows):
                 Use a direct bind mount to a LOCAL filesystem path, NOT a
                 Docker managed volume:
                   volumes:
                     - ./data/db:/app/data     # bind mount — WAL works
                   # NOT: db-data:/app/data    # managed volume — WAL may corrupt

               - CI/CD:
                 Use an ephemeral SQLite in /tmp inside the container (no mount).
                 Tests create + destroy the DB each run.

               - CLOUD DEPLOYMENT (Fly.io, Railway, etc.):
                 Skip SQLite entirely. Use PostgreSQL from day one.
                 Cloud volumes are networked storage — WAL will corrupt.
                 Set DATABASE_URL=postgresql://... in .env for cloud deploys.

               - LINUX NATIVE (no Docker):
                 No issue. WAL works perfectly on native ext4/btrfs.

  ENV-7  [x] Add a startup check in main.py:
               if config.DATABASE_URL.startswith("sqlite") and is_running_in_docker():

DATABASE PROTOCOL (mandatory: prevents raw SQL chaos)
=========================================================================

  ⚠️  VIBE-CODE GUARDRAIL: If the AI reads CREATE TABLE statements, it will
  try to execute raw SQL strings via aiosqlite. This leads to schema drift,
  missing migrations, and impossible-to-debug production issues.

  ONE ORM. ONE MIGRATION TOOL. NO RAW SQL.

  ORM-1  [x] Install SQLAlchemy 2.0 (async) + Alembic:
               pip install sqlalchemy[asyncio] alembic aiosqlite

  ORM-2  [x] Create app/models/base.py:
               from sqlalchemy.orm import DeclarativeBase, MappedAsDataclass
               from sqlalchemy.ext.asyncio import AsyncAttrs

               class Base(AsyncAttrs, DeclarativeBase):
                   pass

  ORM-3  [x] ALL database tables are defined as SQLAlchemy models in app/models/.
              NEVER write raw CREATE TABLE SQL. NEVER call cursor.execute().
              The ONLY SQL that executes is generated by SQLAlchemy and Alembic.

  ORM-4  [x] Example model (labels — replaces the raw CREATE TABLE in Step 3A):

               # app/models/label.py

               from sqlalchemy.orm import Mapped, mapped_column
               from .base import Base

              NEVER modify a table by editing model + restarting. Always migrate.

  ORM-6  [x] Create app/models/__init__.py that imports ALL models:

  ⚠️  BUG PREVENTION: SQLite + async FastAPI concurrency

  SQLite locks the entire database file during writes. Concurrent async
  reads and writes (e.g., background label sync running while users execute

  DB-2   [x] In app/db.py, configure the connection with:
               - PRAGMA journal_mode=WAL  (Write-Ahead Logging — allows

               - PRAGMA busy_timeout=5000  (wait 5s before failing on lock)
               - PRAGMA synchronous=NORMAL  (safe with WAL, better perf)
  DB-3   [x] Use SQLAlchemy async engine with aiosqlite:
               create_async_engine("sqlite+aiosqlite:///./chainscope.db")
  DB-4   [x] Verify: run a write (label insert) and a read (address lookup)

PRECISION STRATEGY (critical: apply to all steps)
=========================================================================

  ⚠️  BUG PREVENTION: Floating point precision loss

  EVM networks denominate in Wei (10^18). Bitcoin uses satoshis (10^8).
  Python float and JavaScript Number will silently drop precision, causing
  corrupted balance calculations and broken min-amount filters.

  PREC-1  [x] Backend (models/schemas.py):
                - All value fields must be Python Decimal, not float
                - Store raw values as strings in JSON responses:
                    "value": "1500000000000000000"  (wei as string)
                    "value_human": "1.5"            (display only)
                    "decimals": 18
                - Never do arithmetic on float-cast blockchain values

  PREC-2  [x] Backend (providers/*.py):
                - Parse all API response values as Decimal immediately:
                    from decimal import Decimal
                    value = Decimal(raw_response["value"])
                - Never: value = float(raw_response["value"])

  PREC-3  [x] Frontend (utils/formatters.js):
                - Use BigInt for all raw value comparisons and arithmetic
                - Use ethers.js parseUnits/formatUnits for human display
                - The min-amount slider must compare BigInt values, not floats

  PREC-4  [x] Frontend (components/TxTable.jsx):
                - Sort-by-amount must sort on raw BigInt, not display string
                - Balance calculations: sum BigInt, format once for display

  PREC-5  [x] Add a test: confirm that 0.1 + 0.2 precision edge case
               doesn't corrupt a balance calculation

  ⚠️  BUG PREVENTION: Recharts / D3.js vs. BigInt Incompatibility

  PREC-6  [x] Recharts relies on D3.js internally for axis domain calculation
               and scale factors. D3 performs math using standard JavaScript
               Number types and will THROW FATAL RENDERING ERRORS if passed
               BigInt objects in data arrays.

               RULE: BigInt is for LEDGER MATH (balances, comparisons, sorting,
               filtering). Standard Number float is for CHART RENDERING.

               In every Recharts data array, explicitly cast BigInt → Number:
                 const chartData = transactions.map(tx => ({
                   ...tx,
                   amount: Number(BigInt(tx.value) / BigInt(10 ** tx.decimals)),
                   // Precision loss is visually imperceptible on a pixel chart.
                   // A 0.0000000001 ETH rendering error is sub-pixel.
                 }))

               Apply this in: TimelineChart.jsx, BalanceChart.jsx,
               StatsPanel.jsx, ExposureChart.jsx — anywhere Recharts consumes
               value data. NEVER use these Number-cast values for balance
               calculations, comparisons, or exports.

  PREC-7  [x] Create a utility in formatters.js:
               function bigIntToChartNumber(rawValue, decimals) {
                 return Number(BigInt(rawValue) / BigInt(10 ** Math.max(decimals - 6, 0)))
                        / (10 ** Math.min(decimals, 6))
               }
               This preserves 6 decimal places of precision — more than enough
               for any chart axis — while safely returning a Number type.

  PREC-8  [x] Test: render a BalanceChart with BigInt values, verify no crash
               and axis labels display correctly

DATA FINALITY STRATEGY (critical: apply to all caching)
=========================================================================

  ⚠️  BUG PREVENTION: Chain Reorganizations & Data Finality

  Since the tool uses public APIs and not full nodes, it is vulnerable to
  chain reorganizations. A transaction may appear as "Success" in an API
  response but get rolled back 60 seconds later when the chain reorgs.
  If you cache that transaction permanently, your DB now contains ghost data.

  FIN-1  [x] Add confirmations field to NormalizedTx schema:
               confirmations: int | None
               Block depth = (current_block - tx_block). If the API doesn't
               return this directly, compute it from the chain's latest block.

  FIN-2  [x] Define finality thresholds (configurable via .env):
               ETH_FINALITY_BLOCKS = 64    (post-Merge finality: ~13 minutes)
               BTC_FINALITY_BLOCKS = 6     (~60 minutes)

  FIN-3  [x] Caching policy:
               - Transactions with confirmations >= finality threshold → cache
                 permanently in DB (safe, will not reorg).
               - Transactions with confirmations < threshold → return to frontend
                 but mark as "unconfirmed". Do NOT write to permanent cache.

DUST FLOOR STRATEGY (critical: apply at provider level)
=========================================================================

  ⚠️  BUG PREVENTION: Dust Attacks Bypassing Spam Filter

  The spam filter (Step 1D) catches zero-value transfers, but "dust attacks"
  send tiny amounts (1 satoshi, 0.000001 ETH) that are technically non-zero.
  These bypass the zero-value check and can overwhelm the BFS trace engine
  with thousands of noise transactions, burning API quota on junk.

  DUST-1  [x] Define configurable dust floors in .env:
                DUST_FLOOR_ETH_WEI = 10000000000000000    (0.01 ETH)
                DUST_FLOOR_BTC_SAT = 10000                (0.0001 BTC)
                # These are PROVIDER-LEVEL floors, applied before data enters
                # the processing pipeline. Different from the user-facing
                # "min_value" filter in the trace engine.

  DUST-2  [x] Implement in providers/base.py as a post-fetch filter:
                def _apply_dust_floor(self, txs: list[NormalizedTx]) -> list[NormalizedTx]:
                    floor = config.dust_floor[self.chain_id()]
                    kept = [tx for tx in txs if Decimal(tx.value) >= floor]
                    dropped = len(txs) - len(kept)
                    if dropped > 0:
                        logger.info(f"Dust floor: dropped {dropped} txs below {floor}")
                    return kept

  DUST-3  [x] Apply AFTER normalization but BEFORE returning to the router.
                This ensures dust never enters the DB cache, the BFS queue,
                or the frontend. The spam_filter (zero-value) runs first,
                then the dust floor catches tiny-but-nonzero transfers.

  DUST-4  [x] Log dust-filtered count in response metadata:

                Frontend shows: "147 dust transactions hidden (below 0.01 ETH)"

  DUST-5  [x] Do NOT apply dust floor to the trace engine's min_value filter.
                The trace min_value is user-controlled and intentional.
                The dust floor is a system-level noise gate.

OUTBOUND RATE LIMITING STRATEGY (critical: apply to all providers)
=========================================================================

  ⚠️  ARCHITECTURAL BOTTLENECK: Proactive vs. Reactive Rate Limiting

  The naive approach — retry with backoff on 429 — fails under concurrency.
  If multiple users or batch processes fire lookups simultaneously, they will
  all slam into Etherscan's 5 calls/sec ceiling, triggering cascading 429
  errors and exponential backoffs. This leads to API ban-hammering.

  SOLUTION: A proactive, global Token Bucket that throttles outbound requests
  BEFORE they are sent. A 429 should practically never be triggered.

  RATE-1  [x] Install aiolimiter (async token bucket implementation)

  RATE-2  [x] Create app/outbound_limiter.py:

                from aiolimiter import AsyncLimiter

                LIMITERS = {
                  "etherscan":   AsyncLimiter(4.5, 1),   # 4.5 req/sec (limit is 5)
                  "bscscan":     AsyncLimiter(4.5, 1),
                  "polygonscan": AsyncLimiter(4.5, 1),
                  "blockstream": AsyncLimiter(8.0, 1),   # ~10 req/s soft limit
                  "helius":      AsyncLimiter(8.0, 1),   # 10 req/s limit
                  "trongrid":    AsyncLimiter(12.0, 1),  # 15 req/s limit
                  "coingecko":   AsyncLimiter(0.4, 1),   # ~25/min = 0.4/sec
                  # 4byte: NOT HERE — uses bulk download, not live API (see Step 1H)
                }

  RATE-3  [x] In providers/base.py, wrap every outbound HTTP call:

                async def _rate_limited_request(self, provider_name, url, params):
                    limiter = LIMITERS[provider_name]
                    async with limiter:
                        return await self._http_client.get(url, params=params)

  RATE-4  [x] The @retry decorator now ONLY handles genuine errors (5xx, timeouts),

--- 1E: BTC Change Detection ---

  ⚠️  BUG PREVENTION: Flawed Change Heuristics

  The naive "largest output to known address" is incorrect. Change goes to
  NEW addresses. Additionally, "script type mismatch" is increasingly
  unreliable in the SegWit/Taproot era — modern wallets (Ledger, Trezor)
  routinely mix script types (spending from 1... to bc1... change).

  1E-1  [x] Create services/change_detect.py with REPRIORITIZED heuristics:

              def detect_change_output(tx) → ChangeDetection | None:

              PRIORITY 1 — First-time-seen address (STRONGEST signal):
                If one output goes to an address with zero prior transaction
                history, it is very likely a freshly generated change address.
                This is the strongest heuristic because wallets generate new
                change addresses by default for privacy.
                Implementation: check cache first, then optionally make one
                API call to Blockstream GET /api/address/{addr} and check
                tx_count == 0. Cache the result.
                IMPORTANT: This is now the primary heuristic, not a supplement.

              PRIORITY 2 — Round number analysis (GOOD signal):
                Payments tend to be round numbers (1.0, 0.5, 0.001 BTC).
                Change is the messy remainder (0.04837291 BTC).
                If exactly one output is non-round and the rest are round,
                the non-round output is likely change.

              PRIORITY 3 — Script type matching (WEAK signal, use with caution):
                If inputs and one output share the same script type, that output
                MAY be change. BUT: modern wallets mix types freely (SegWit →
                Taproot change). Only use this as a TIE-BREAKER when Heuristics
                1 and 2 are inconclusive. Never rely on it alone.

              Confidence scoring:
                Heuristic 1 match alone         → HIGH confidence
                Heuristic 1 + 2 agree           → HIGH confidence
                Heuristic 2 match alone          → MEDIUM confidence
                Heuristic 3 match alone          → LOW confidence (barely useful)
                Conflicting signals              → LOW confidence
                Return: { output_index, confidence, reasons: [], heuristics_used: [] }

  The 4byte.directory API does NOT support bulk queries — you cannot send
  an array of 50 selectors in one request. If a heavy DeFi contract has 40
  unknown function selectors, that's 40 sequential HTTP requests. At the
  token bucket limit of 2.0 req/s, this stalls the lookup for 20 seconds,
  causing a timeout.

  The entire 4byte signature database is remarkably small (~5MB as CSV).

  instant local SQLite queries.

  ⚠️  BUG PREVENTION: SSE Payload Saturation (Memory Bloat)

  5B-2  [x] SSE events must stream DELTAS ONLY — never the full graph.

              If the trace discovers 500 nodes and 1,000 edges, pushing the
              entire graph structure on every iteration will saturate the
              browser network tab and choke React's memory heap. The payload
              grows quadratically with graph size.

              RULE: The backend pushes only the NEWLY DISCOVERED data per event.
              The frontend (useTraceStream.js) is responsible for appending
              deltas to the master graph state in React.

              SSE event types (ALL deltas, never full snapshots):

              event: progress
              data: { "hop": 1, "max_hops": 3, "nodes_found": 12,
                      "edges_found": 23, "api_calls_used": 15,
                      "api_calls_limit": 200 }
              // Lightweight counters only. No node/edge data.

              event: node_discovered
              data: { "address": "0x...", "label": "Binance", "risk": "LOW",
                      "hop": 1 }
              // Single node. Frontend appends to nodes array.

              event: edge_discovered
              data: { "from": "0x...", "to": "0x...", "value": "150000...",
                      "tx_hash": "0x...", "token": "ETH", "timestamp": 1711... }
              // Single edge. Frontend appends to edges array.
              // Batching: if multiple edges discovered in one BFS step,
              // send up to 10 edges per SSE event as an array to reduce
              // event overhead. Never more than 10 per event.

              event: pruned
              data: { "address": "0x...", "reason": "exchange", "hop": 2 }

              event: sampled
              data: { "address": "0x...", "total_txs": 2341, "sampled": 50 }

              event: peeling_chain_detected
              data: { "chain_length": 14, "total_peeled": "2.3 BTC", ... }

              event: warning
              data: { "message": "Etherscan tokentx timed out for 0xabc..." }

              event: completed
              data: { "total_nodes": 147, "total_edges": 289,
                      "trace_time_ms": 4200,
                      "pruned_at": [...], "sampling": [...] }
              // Metadata ONLY. No node/edge data. The frontend already has
              // the full graph from accumulated deltas.

              event: failed
              data: { "error": "...", "partial": true }
              // Frontend uses whatever deltas it accumulated so far.

  5E-1  [x] Build TraceControls.jsx (direction, hops, min_value, pruning, sampling)
  5E-2  [x] Implement SSE client in hooks/useTraceStream.js:

              ⚠️  BUG PREVENTION: React State Thrashing on SSE Deltas (UI Freeze)

              Even with useRef, calling React state updates on every SSE event
              will trigger re-renders of the graph component for each of the
              500+ nodes and 1000+ edges. This throttles the main thread,
              freezes the UI, and makes the cancel button unresponsive.

              SOLUTION: Completely bypass React's render cycle for graph topology
              updates. Use an imperative handle to push deltas directly into the
              Cytoscape or Sigma instance. React state is updated on a slow
              cadence (1s tick) ONLY for text-based counters (TraceProgress).

              ARCHITECTURE (two data channels):

              Channel 1: IMPERATIVE (fast, no React re-render)
                SSE events → useTraceStream → graphRef.current.addNode(node)
                                             → graphRef.current.addEdges(batch)
                The graph renderer exposes an imperative API via forwardRef:
                  <GraphView ref={graphRef} ... />
                  graphRef.current = {
                    addNode(nodeData),     // directly calls cy.add() or sigma.addNode()
                    addEdges(edgeBatch),   // directly calls cy.add() or sigma.addEdge()
                    getGraph(),            // returns current {nodes, edges} for export
                  }
                These methods mutate the graph instance directly — zero React involvement.

              Channel 2: THROTTLED REACT STATE (slow, 1s tick, for UI counters)
                SSE events → accumulate counters in a plain JS object (not state)
                Every 1 second (setInterval), flush counters to React state:
                  setProgress({ nodes: counter.nodes, edges: counter.edges,
                                hop: counter.hop, apiCalls: counter.apiCalls })
                TraceProgress.jsx re-renders once per second — smooth and responsive.

              Implementation:

              const useTraceStream = (graphRef) => {
                // Counters — plain JS, NOT React state
                const counters = useRef({ nodes: 0, edges: 0, hop: 0, apiCalls: 0 })

                // Throttled React state for UI — updated every 1s
                const [progress, setProgress] = useState(null)
                const [metadata, setMetadata] = useState(null)
                const [status, setStatus] = useState("idle") // idle|streaming|completed|failed

                useEffect(() => {
                  // Flush counters to React state every 1 second
                  const interval = setInterval(() => {
                    setProgress({ ...counters.current })
                  }, 1000)
                  return () => clearInterval(interval)
                }, [])

                const startTrace = async (params) => {
                  const { job_id, stream_url } = await fetch("/api/trace", { ... })
                  setStatus("streaming")
                  const source = new EventSource(stream_url)

                  source.addEventListener("node_discovered", (e) => {
                    const node = JSON.parse(e.data)
                    counters.current.nodes++
                    // IMPERATIVE: push directly into graph renderer — no React
                    graphRef.current?.addNode(node)
                  })

                  source.addEventListener("edge_discovered", (e) => {
                    const batch = JSON.parse(e.data)
                    counters.current.edges += batch.length
                    // IMPERATIVE: push directly into graph renderer — no React
                    graphRef.current?.addEdges(batch)
                  })

                  source.addEventListener("progress", (e) => {
                    const p = JSON.parse(e.data)
                    counters.current.hop = p.hop
                    counters.current.apiCalls = p.api_calls_used
                    // NOT calling setState here — the 1s interval handles it
                  })

                  source.addEventListener("completed", (e) => {
                    setMetadata(JSON.parse(e.data))
                    setStatus("completed")
                    setProgress({ ...counters.current }) // final flush
                    source.close()
                  })

                  source.addEventListener("failed", (e) => {
                    setStatus("failed")
                    source.close()
                  })
                }

                return { progress, metadata, status, startTrace, cancelTrace }
              }

  5E-3  [x] Build GraphView.jsx with forwardRef imperative handle:
              const GraphView = forwardRef((props, ref) => {

--- 7B: Rendering Engine ---

  ⚠️  BUG PREVENTION: Renderer Auto-Switch Race Condition (Crash Risk)

  The original plan auto-switches from Cytoscape (<300 nodes) to Sigma
  (300+ nodes) based on current node count. During a live SSE trace, this
  is CATASTROPHIC: at node 299 Cytoscape is rendering. Node 300 arrives.
  React unmounts the entire Cytoscape canvas, destroys the DOM, mounts
  a Sigma WebGL canvas, requests a new Web Worker layout for 300 nodes,
  while SSE events 301/302/303 are still arriving. This causes a
  rendering loop and crashes the browser tab.

  SOLUTION: Lock the renderer BEFORE the trace starts. Never switch
  renderers while a trace is streaming or a graph is displayed.

  7B-1  [ ] Create hooks/useGraphRenderer.js with LOCKED renderer selection:
              - The renderer is chosen ONCE and locked for the session:
                  Option A: User manually selects in TraceControls.jsx:
                    "Graph Size: Small (< 300 nodes) | Large (300+ nodes)"
                    Small → Cytoscape initialized at node 0
                    Large → Sigma initialized at node 0

                  Option B: Auto-select based on trace parameters:
                    If max_hops >= 3 AND trace_through_entities === true → Sigma
                    Otherwise → Cytoscape
                    Lock at trace start, never re-evaluate.

              - The renderer component is NEVER unmounted while data exists.
              - If the user wants to switch renderers after a trace completes:
                provide a "Switch to Sigma/Cytoscape" button that:
                  1. Copies current graph state to a buffer
                  2. Unmounts old renderer (cleanly, no race)
                  3. Mounts new renderer
                  4. Injects buffered state
                  5. Re-runs layout via Web Worker
                This is a deliberate user action, not an automatic threshold.

  7B-2  [ ] Both renderers receive pre-computed positions from Web Worker
  7B-3  [ ] Install cytoscape (NOT cytoscape-dagre), sigma + graphology
  7B-4  [ ] Build GraphView.jsx wrapper:

BscScan              Yes     5 calls/sec       4.5 req/s
PolygonScan          Yes     5 calls/sec       4.5 req/s
Arbiscan             Yes     5 calls/sec       4.5 req/s
CoinGecko            Opt     ~25/min           0.4 req/s
CoinGecko (hist)     Opt     ~10/min           0.15 req/s  (historical endpoint)
Helius (Solana)      Yes     10 req/s          8.0 req/s
TronGrid             Yes     15 req/s          12.0 req/s
4byte.directory      No      N/A               N/A (bulk download, no live API)
LayerZero scan       Check   Check docs        Set after testing
Wormhole scan        Check   Check docs        Set after testing

Strategy:
  - Global token bucket per provider (proactive, not reactive)
  - Cache transactions forever once finalized. Unfinalized: 10min TTL.
  - Historical prices: cache forever (price on a past date never changes).
  - Method signatures: cache forever (selectors are deterministic).
  - Batch label endpoint reduces lookups.
  - Trace engine: budget calls across hops (40/35/25 split).
  - max_txs_per_node prevents single-node quota burn.
  - Dust floor prevents noise from entering pipeline at all.
  - Monitor via GET /api/admin/rate-status.
  - 429 = bug → log as WARNING.
